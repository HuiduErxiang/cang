## 0. 方案目的

本文解决的核心问题是：

**在现有蒸馏链（来源入库 → 蒸馏 → 发布）之上，增加一层"进货与分仓机制"，确保进入知识仓的资料经过统一接收、盘点、去重、分类后，再按类型落位到不同的原始仓。**

当前痛点：

1. **没有统一入口**：资料来源多样（外部导入、手工录入、迁移遗留），入口散乱
2. **没有盘点环节**：资料进入后无法追溯来源、无法判断是否重复
3. **分类规则缺失**：`raw/` 下的组织方式不一致（有按疾病分，有按来源分）
4. **血缘不完整**：从"投递文件"到"正式知识"的链路断开

## 1. 分仓目标

### 1.1 为什么需要"总进货仓 + 盘点 + 分发到 raw 子仓"？

**直接把资料丢进 `raw/` 的问题：**

| 问题 | 表现 |
|------|------|
| **无法溯源** | 不知道资料是谁投递的、什么时候投递的、原始文件名是什么 |
| **无法去重** | 同一份 PDF 可能被多次导入，产生冗余副本 |
| **分类混乱** | 投递者可能按自己习惯命名和归类，与知识仓规范冲突 |
| **元数据缺失** | 文件进入时没有统一的元数据补充流程 |
| **责任不清** | 资料质量问题时，无法追溯到投递者或投递批次 |

**引入"总进货仓 + 盘点 + 分发"的价值：**

| 环节 | 价值 |
|------|------|
| **统一入口** | 所有资料必须经过 `intake/` 进入，确保可追踪 |
| **盘点环节** | 识别文件类型、生成唯一标识、计算内容哈希、与已有库存比对 |
| **去重机制** | 通过内容哈希识别重复文件，避免冗余入库 |
| **元数据补充** | 在盘点阶段统一补充来源、分类、可信度等元数据 |
| **分类分发** | 根据盘点结果，将资料分发到正确的 `raw/` 子仓 |

**一句话：`intake/` 是知识仓的"海关"，`raw/` 是"保税仓"，没有经过海关的货物不能直接进入保税仓。**

### 2.2 为什么这样设计目录？

**`intake/inbox/` 的三种入口：**

| 入口 | 使用场景 | 特点 |
|------|----------|------|
| `drop/` | 文件系统直接投递 | 简单粗暴，适合批量迁移 |
| `upload/` | API/界面投递 | 可携带元数据，适合日常操作 |
| `import/` | 批量迁移导入 | 带来源清单，适合从旧系统迁移 |

**`raw/` 的分仓维度选择：**

我选择**按"来源渠道"分仓**，而非按内容类型、疾病领域或业务主题分仓。原因：

| 维度 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| **来源渠道** | 盘点时可从投递信息直接判断，无需分析文件内容 | 同一渠道内资料类型可能多样 | ✅ 采用 |
| **内容类型** | 符合资料处理习惯 | 判断需要分析文件内容，可能与来源渠道交叉 | ❌ 留给子目录或蒸馏环节 |
| **疾病领域** | 符合医生思维习惯 | 来源渠道混合，可信度难以区分 | ❌ 不采用 |
| **业务主题** | 贴近应用场景 | 主题边界模糊，分类困难 | ❌ 不采用 |

**⚠️ 重要：分仓规则采用"优先级仲裁"机制**

当资料同时满足多个规则条件时，按优先级从高到低仲裁，确保最终落入唯一仓：
1. 第一优先级：来源系统标记（最可靠）
2. 第二优先级：投递方式
3. 第三优先级：投递者角色
4. 第四优先级：迁移标记
5. 兜底：orphan（需人工确认）

**核心原则：`raw/` 按"来源渠道"分仓，通过优先级仲裁确保每份资料落入唯一仓。内容类型/疾病/主题分类留给蒸馏环节。**

### 3.2 各环节详细说明

#### 阶段 1：投递（Deposit）

**投递方式：**

| 方式 | 入口目录 | 携带信息 | 适用场景 |
|------|----------|----------|----------|
| **直接投递** | `intake/inbox/drop/` | 仅文件 | 批量迁移、快速导入 |
| **上传投递** | `intake/inbox/upload/` | 文件 + 元数据 JSON | API 调用、界面操作 |
| **导入投递** | `intake/inbox/import/` | 文件 + 批次清单 | 系统迁移、批量导入 |

**投递时生成的记录：**

```json
// intake/manifests/intake_registry.json 示例
{
  "intake_id": "INT-20260316-0001",
  "status": "pending_review",
  "deposited_at": "2026-03-16T10:30:00Z",
  "deposited_by": "user_zhang",
  "deposit_method": "upload",
  "original_filename": "仑卡奈单抗中国真实世界研究.pdf",
  "file_size_bytes": 2048576,
  "content_hash": "a1b2c3d4e5f6...",
  "inbox_path": "intake/inbox/upload/INT-20260316-0001.pdf",
  "metadata": {
    "suggested_category": "clinical_trials",
    "source_system": "医学数据库",
    "tags": ["lecanemab", "AD", "真实世界"]
  }
}
```

#### 阶段 2：盘点（Inventory）

**盘点操作清单：**

| 步骤 | 操作 | 输出 |
|------|------|------|
| **1. 文件识别** | 解析 MIME 类型、文件格式 | `mime_type`, `file_format` |
| **2. 唯一标识生成** | 生成 `source_key` | `SRC-EVIDENCE-20260316-0001` |
| **3. 内容哈希计算** | SHA-256 截取前 16 位 | `content_hash` |
| **4. 去重检查** | 与 `dedup_index.json` 比对 | `is_duplicate`, `duplicate_of` |
| **5. 元数据补充** | 根据文件内容推断 | 分类、可信度、来源等 |
| **6. 分类判断** | 根据规则确定目标仓 | `target_warehouse` |

**盘点后状态：**

#### 阶段 3：分发（Dispatch）

**分发规则（按来源渠道）：**

| 来源渠道 | 目标仓 | 判断依据 |
|----------|--------|----------|
| 用户上传 | `raw/deposited/user_upload/` | `deposit_method = 'upload'` |
| 批量导入 | `raw/deposited/batch_import/` | `deposit_method = 'import'` |
| 系统投递 | `raw/deposited/system_drop/` | `deposit_method = 'drop'` |
| 医学数据库导入 | `raw/external/medical_database/` | `source_system = '医学数据库'` |
| 合作方共享 | `raw/external/partner_shared/` | `source_system` 在合作方列表 |
| 网络抓取 | `raw/external/web_crawled/` | `source_system = '网络爬虫'` |
| 编辑部产出 | `raw/internal/editorial/` | `deposited_by_role = 'editor'` |
| 专家录入 | `raw/internal/expert_input/` | `deposited_by_role = 'expert'` |
| 会议纪要 | `raw/internal/meeting_notes/` | 文件名包含"会议纪要" |
| V2 历史资料 | `raw/migrated/v2_historical/` | `source_system = 'V2迁移'` |
| 其他遗留 | `raw/migrated/legacy_import/` | `source_system` 标记为遗留 |
| 来源不明 | `raw/orphan/` | 无法判断来源渠道 |

**说明：内容类型（临床试验/指南/文献等）不再作为分仓维度，改为在 SourceRecord 的元数据中标记，供蒸馏环节使用。**

**分发后的处理：**

1. 文件从 `intake/reviewed/` 移动到 `raw/<warehouse>/`
2. 在 `raw/` 下生成索引记录
3. 更新 `lineage/source_index.json`，建立血缘起点

## 4. 分仓规则

### 4.1 分仓维度选择

**采用"来源渠道"作为唯一分仓维度，原因：**

1. **盘点阶段可判断**：来源渠道从投递信息直接获取，无需分析文件内容
2. **决策机制明确**：允许多命中，但按固定优先级仲裁确定唯一仓（见 4.2 规则表）
3. **与内容类型解耦**：内容类型（临床试验/指南/文献等）在元数据标记，不参与分仓

### 4.2 分类规则表

**分仓规则（按来源渠道，优先级从高到低）：**

```yaml
classification_rules:
  # 第一优先级：来源系统标记（最可靠）
  external:
    medical_database:
      trigger: "source_system == '医学数据库'"
      target: "raw/external/medical_database/"
    
    partner_shared:
      trigger: "source_system in ['合作方', '甲方', '药企']"
      target: "raw/external/partner_shared/"
    
    web_crawled:
      trigger: "source_system == '网络爬虫'"
      target: "raw/external/web_crawled/"
  
  # 第二优先级：投递方式
  deposited:
    user_upload:
      trigger: "deposit_method == 'upload'"
      target: "raw/deposited/user_upload/"
    
    batch_import:
      trigger: "deposit_method == 'import'"
      target: "raw/deposited/batch_import/"
    
    system_drop:
      trigger: "deposit_method == 'drop'"
      target: "raw/deposited/system_drop/"
  
  # 第三优先级：投递者角色
  internal:
    editorial:
      trigger: "deposited_by_role in ['editor', '编辑']"
      target: "raw/internal/editorial/"
    
    expert_input:
      trigger: "deposited_by_role in ['expert', '医学专家']"
      target: "raw/internal/expert_input/"
    
    meeting_notes:
      trigger: "filename contains '会议纪要' or '会议记录'"
      target: "raw/internal/meeting_notes/"
  
  # 第四优先级：迁移标记
  migrated:
    v2_historical:
      trigger: "source_system == 'V2迁移'"
      target: "raw/migrated/v2_historical/"
    
    legacy_import:
      trigger: "source_system contains 'legacy' or '遗留'"
      target: "raw/migrated/legacy_import/"
  
  # 兜底：无法判断来源渠道
  orphan:
    trigger: "default"
    target: "raw/orphan/"
    description: "无法判断来源渠道时落入此仓，需人工确认"
```

### 4.3 内容类型标记（不分仓）

内容类型不再参与分仓，改为在**蒸馏环节**识别并记录到元数据：

```yaml
content_type_tags:  # 由蒸馏环节识别，仅作为元数据标记，不影响分仓
  - CLINICAL_TRIAL   # 临床试验
  - GUIDELINE        # 指南
  - CONSENSUS        # 共识
  - REFERENCE        # 参考文献
  - CONFERENCE       # 会议资料
  - REAL_WORLD       # 真实世界研究
  - UNKNOWN          # 未识别
```

**说明**：进货阶段只产出来源渠道判断，内容类型需分析文件内容，属于蒸馏环节职责。

### 4.4 分类置信度

当自动分类置信度低于 0.7 时，资料进入 `raw/orphan/`，等待人工确认。

### 5.2 对象定义

#### IntakeRecord（进货记录）

```typescript
interface IntakeRecord {
  // 唯一标识
  intake_id: string;           // INT-{YYYYMMDD}-{序号}
  
  // 投递信息
  deposited_at: string;        // ISO 时间戳
  deposited_by: string;        // 投递者标识
  deposit_method: 'drop' | 'upload' | 'import';
  original_filename: string;
  original_path?: string;      // 原始来源路径
  
  // 文件信息
  file_size_bytes: number;
  content_hash: string;        // SHA-256 前 16 位
  mime_type: string;
  
  // 状态
  status: IntakeStatus;
  
  // 盘点结果
  inventory_result?: InventoryResult;
  
  // 分类结果
  classification_result?: ClassificationResult;
  
  // 关联
  source_key?: string;         // 关联的来源记录
  batch_id?: string;           // 所属批次
}

type IntakeStatus = 
  | 'pending_review'    // 待盘点
```

## 7. 与现有层级的关系

### 7.1 职责边界

| 层级 | 职责 | 不负责 |
|------|------|--------|
| **intake/** | 接收、盘点、去重、分类、分发 | 存储、蒸馏、发布 |
| **raw/** | 存储原始来源，保留原始形态 | 内容处理、结构化 |
| **staging/** | 蒸馏中间产物、片段提取、事实构建 | 原始存储、正式发布 |
| **l1~l4/** | 正式知识层，晋升后的稳定知识 | 原始资料、中间产物 |
| **lineage/** | 血缘记录、状态追踪、审计日志 | 内容存储 |
| **publish/** | 对外发布快照 | 内部处理 |

### 7.2 数据流向

```
intake/ → raw/ → staging/ → l1~l4/ → publish/
   │        │        │          │         │
   └────────┴────────┴──────────┴─────────┘
                       │
                       ▼
                   lineage/
              (全程血缘追踪)
```

### 7.3 关键原则

1. **intake/ 是海关，不是仓库**：资料不应在 intake/ 长期驻留
2. **raw/ 是保税仓，不是货架**：不直接对外提供，仅供内部蒸馏
3. **staging/ 是车间，不是展厅**：中间产物，不保证稳定
4. **l1~l4/ 是正式库，直接面向发布**
5. **publish/ 是橱窗，消费者只看这里**

## 9. 风险与边界

### 9.1 最容易做错的点

| 风险 | 表现 | 后果 | 预防措施 |
|------|------|------|----------|
| **把 intake 当正式知识层** | 直接从 intake/ 读取知识 | 数据不一致、无法追踪 | 明确 intake/ 不对外提供 |
| **把 raw 当发布层** | 消费系统直接读 raw/ | 内部结构暴露、耦合过紧 | 消费系统只能读 publish/ |
| **分类规则过早复杂化** | 试图在进货阶段完成所有分类 | 规则难以维护、错误率高 | 分类只做到来源渠道，细分留给蒸馏 |
| **忽略去重** | 不做内容哈希比对 | 大量重复入库 | 强制去重检查，重复文件自动跳过 |
| **元数据缺失** | 进货时不补充元数据 | 后续处理困难 | 盘点阶段必须补充基础元数据 |
| **血缘断链** | 只记录部分血缘 | 无法追溯来源 | 全链路血缘，从投递到发布 |
| **状态混乱** | 状态定义不清晰、流转无记录 | 无法知道资料处于什么阶段 | 明确状态定义、记录所有状态变更 |

### 9.2 边界约束

| 边界 | 约束 |
|------|------|
| **时间边界** | 盘点应在投递后 24 小时内完成 |
| **存储边界** | intake/ 不长期存储，资料应在盘点后 72 小时内分发 |
| **分类边界** | 进货分类只到来源渠道，不涉及疾病/主题/产品 |
| **元数据边界** | 进货阶段只补充基础元数据，深度提取留给蒸馏 |
| **血缘边界** | 血缘记录不可删除，只能追加 |
