> 口径：只允许使用 `Implemented / Verified / Still Open`

---

## 1. 适用范围

适用于：

1. 几十到几百份 PDF 的批量导入
2. 导入后继续批量蒸馏
3. 最终要形成可检索、可回查、可继续发布的结构化产物

不适用于：

1. 单篇人工临时阅读
2. 只想把文件拷进目录，不做蒸馏
3. 只做 OCR / 抽文本

## 4. 硬约束

1. 文件入仓，不等于知识入仓。
2. 文件不进 `raw/pdf/`，不得进入蒸馏。
3. PDF 蒸馏主路径固定为：原生多模态模型直读原始 PDF / 页面。
4. OCR、抽文本、pdfplumber 一类方案，不得写成主路线。
5. 全量蒸馏前，必须先做“每类至少 3 份样本验证”。
6. 样本验证不过，不得直接批跑几百份。
7. 蒸馏产物必须可 `json.load()`。
8. 蒸馏产物必须带文献身份信息、原 PDF 路径、页码锚点。
9. 不得把“样本通过”写成“全量蒸馏完成”。
10. 不得把“文件在 raw/pdf/”写成“侠客岛已可直接消费”。
11. 大体积 PDF 必须单独处理，不得与常规 PDF 混跑。

---

## 5. 总流程

顺序固定：

1. 冻结源目录与数字口径
2. 批量入仓到 `raw/pdf/`
3. 写 `batch manifest`
4. 写 `lineage/source index`
5. 做四类样本蒸馏验证
6. 样本通过后启动全量批蒸馏
7. 对失败件重试 / 降级分批
8. 产出最小检索索引
9. 回写 checkpoint
10. 清理过程垃圾

## 6. Phase A：批量入仓 SOP

### A1. 冻结输入

必须先写死：

1. 源目录
2. 只处理的文件类型
3. 当前数字口径
4. 分类映射规则

未冻结不得开跑。

### A2. 盘点数字

必须统计：

1. 源总数
2. 重复文件名数
3. 唯一文件名数
4. 藏经阁原有数
5. 新增数
6. 导入后总数

不得同时出现多套数字。

### A3. 复制入仓

要求：

1. 文件实体进入 `raw/pdf/...`
2. 原文件名默认保留
3. 相对路径稳定
4. 目录分类可解释

禁止：

1. 只写清单不拷文件
2. 直接跳到 L3/L4
3. 平铺所有 PDF 到一个目录

### A4. 写入 manifest

至少落：

1. `intake/manifests/intake_batches/<batch>.json`

每条至少有：

1. `filename`
2. `relative_path`
3. `category`
4. `size_bytes`
5. `seq`

### A5. 写入 lineage

至少落：

1. `lineage/source_index_<date>.json`

至少包含：

1. `total_pdfs`
2. `categories`
3. 分类到文件名的最小索引

### A6. 入仓验收线

满足以下条件才算 `Verified`：

1. `raw/pdf/` 真实文件数与结论文档一致
2. `batch manifest` 记录数与结论文档一致
3. `source index` 总数与结论文档一致
4. 没有冲突数字

## 7. Phase B：样本蒸馏验证 SOP

### B1. 样本规则

必须按文档类型抽样，每类至少 3 份：

1. `original_research`
2. `guideline_consensus`
3. `epidemiology`
4. `conference`

样本不足 3 份时，全做。

### B2. 蒸馏规则

必须使用：

1. 原生多模态模型直读 PDF / 页面
2. 结构化 JSON 输出
3. 稳定管道写盘

### B3. 样本 JSON 最低字段

公共字段：

1. `title`
2. `source`
3. `year`
4. `document_type`
5. `language`
6. `total_pages`
7. `doi`（如有）
8. `_source_pdf`

类型字段：

1. 研究原文：`trial_name / study_design / sample_size / primary_endpoint`
2. 指南：`issuing_body / disease_area / key_recommendations`
3. 流行病学：`data_source / time_period / geography`
4. 会议：`conference_name / session_type / key_studies`

锚点字段：

1. `page`
2. `figure`
3. `table`
4. `caption`

### B4. 样本验收线

全部满足才算 `Verified`：

1. `json.load()` 通过
2. 四类样本都过线
3. 文献身份能分清是哪一版
4. 关键结论至少能回到原 PDF 路径 + 页码
5. 至少形成一套最小检索索引

样本不过，不得进入全量批跑。

---

## 8. Phase C：全量蒸馏 SOP

### C1. 启动条件

只有以下条件全部满足，才允许全量启动：

1. Phase A 已 `Verified`
2. Phase B 已 `Verified`
3. 有失败重试方案
4. 有超时处理方案
5. 有 checkpoint 回写机制

### C2. 运行原则

“一口气把几百个 PDF 全部蒸馏进去”在执行上指的是：

1. 同一轮任务内完成全量批处理
2. 可以分批调用模型
3. 不等于单次调用处理几百份

### C3. 批跑要求

必须有：

1. 总批次表
2. 成功清单
3. 失败清单
4. 重试清单
5. 最终完成率

### C4. 大文件处理

大体积 PDF 必须单独队列处理。

默认判定线：

1. 文件体积 `>10MB`
2. 或页数显著偏大
3. 或样本阶段已出现超时 / 截断 / 锚点缺失

对大体积 PDF 默认执行：

1. 延长超时
2. 必要时分页
3. 必要时分章节
4. 单独记录成功率与失败率

不得因为“大文件难跑”就整体跳过某一类文档。
不得把“大体积 PDF 未处理”混写成“全量蒸馏完成”。

### C5. 全量验收线

只有全部满足才允许写“全量蒸馏完成”：

1. 全量 PDF 都有蒸馏结果或明确失败记录
2. 失败件数量明确
3. 每份 JSON 可解析
4. 每份 JSON 可回到原 PDF
5. 检索索引覆盖全部已蒸馏文献

否则只能写：

1. `部分完成`
2. `失败件待补`
3. `Still Open`

---

## 9. 最小检索 SOP

蒸馏后至少要产出一套最小索引，支持：

1. 按研究名找 PDF
2. 按 trial_name 找 PDF
3. 按关键词找 PDF
4. 返回原 PDF 相对路径
5. 返回关键页码 / figure / table 锚点

没有这套索引，不得写“可按需查证”。
